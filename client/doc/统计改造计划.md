# 统计改造计划

## 背景

当前 Analytics 看板存在两个核心问题：

1. 页面顶部只有一个全局时间范围，但概览页混合展示了长期统计、固定窗口统计和可选时间统计，例如总客户端数、今日活跃、近 7 天活跃、近 30 天活跃、每日统计等，查看者很难判断某个指标是否受顶部时间范围影响。
2. 核心统计主要依赖 Cloudflare Workers Analytics Engine，Analytics Engine 数据保留期约 3 个月，不能承担长期累计统计。总客户端数、累计访问、累计模型请求、资源累计点击等指标如果只从 Analytics Engine 查询，长期会丢失。

本计划将统计系统拆成“长期统计、固定时间统计、可选时间统计”三类，并新增 D1 长期预聚合存储。D1 不保存原始事件明细，不在页面查询时扫描大范围事件，而是保存客户端身份、轻量活跃索引和按月预聚合结果。

## 改造目标

1. 统计页面上明确区分长期、固定窗口、可选范围，避免一个全局时间选择影响不一致。
2. 需要长期保存的核心统计写入 D1，避免 Analytics Engine 保留期导致数据丢失。
3. 访问分析、配置使用、模型使用、资源管理支持“历史总数”维度。
4. D1 只保存预聚合结果和必要去重索引，不保存原始事件全文，避免 D1 成为日志库。
5. `/track` 上报不因 D1 写入而明显变慢，长期统计写入通过 Queue 异步批处理。
6. 指标口径、数据源和时间范围在 UI 与 README 中保持一致。

## 总体架构

```text
客户端埋点
  -> /track Worker
    -> Analytics Engine：保留原始事件，用于最近事件、90 天内灵活分析、临时排查
    -> Cloudflare Queue：投递规范化事件
      -> Queue Consumer
        -> ANALYTICS_DB D1：客户端身份、每日活跃索引、月度预聚合、历史唯一客户端计数

Dashboard
  -> 长期累计 / 固定窗口 / 历史总数：读 D1
  -> 最近 7/30/90 天灵活分析：优先读 Analytics Engine，必要时读 D1 每日/月度聚合
```

## 数据口径分层

| 类型 | 是否受页面时间选择影响 | 数据源 | 说明 |
| --- | --- | --- | --- |
| 长期统计 | 不受影响 | D1 | 从长期化启用日起持续累计，可回填最近 90 天 |
| 固定窗口统计 | 不受可选范围影响 | D1 | 今日、昨日、近 7 天、近 30 天等固定口径 |
| 可选时间统计 | 只受当前模块自己的时间选择影响 | Analytics Engine 或 D1 | 最近 7/30/90 天趋势、排行、分布 |
| 历史总数 | 不受时间选择影响 | D1 月度聚合 | 访问分析、配置使用、模型使用、资源管理的长期累计 |

## D1 数据库设计

新增独立 D1 数据库绑定：`ANALYTICS_DB`。不要复用 `RESOURCE_DB`。

### 时间与日期口径

所有原始时间戳保存为 UTC ISO 字符串，所有用于统计分组的 `date`、`month` 字段统一使用业务时区 `Asia/Shanghai` 计算。

| 字段类型 | 口径 |
| --- | --- |
| `received_at`、`first_seen_at`、`last_seen_at`、`processed_at` | UTC ISO 时间戳 |
| `activity_date`、`first_seen_date`、`last_seen_date`、`client_created_date` | `Asia/Shanghai` 日期，格式 `YYYY-MM-DD` |
| `month`、`first_seen_month` | `Asia/Shanghai` 月份，格式 `YYYY-MM` |

Dashboard 中的“今日”“昨日”“近 7 天”“近 30 天”全部按 `Asia/Shanghai` 计算。不得直接使用 Worker 所在环境的默认时区，也不得混用 UTC 日期作为业务日期。

### 客户端身份表

```sql
CREATE TABLE IF NOT EXISTS analytics_clients (
  project_name TEXT NOT NULL,
  client_id TEXT NOT NULL,
  reported_client_created_date TEXT,
  client_created_date TEXT NOT NULL,
  client_created_source TEXT NOT NULL,
  first_seen_at TEXT NOT NULL,
  first_seen_date TEXT NOT NULL,
  first_seen_source TEXT NOT NULL,
  last_seen_at TEXT NOT NULL,
  last_seen_date TEXT NOT NULL,
  last_seen_source TEXT NOT NULL,
  first_version TEXT NOT NULL DEFAULT '',
  last_version TEXT NOT NULL DEFAULT '',
  platform TEXT NOT NULL DEFAULT '',
  arch TEXT NOT NULL DEFAULT '',
  PRIMARY KEY (project_name, client_id)
);

CREATE INDEX IF NOT EXISTS idx_analytics_clients_project_last_seen
ON analytics_clients (project_name, last_seen_date);

CREATE INDEX IF NOT EXISTS idx_analytics_clients_project_created
ON analytics_clients (project_name, client_created_date);
```

用途：长期总客户端、最近活跃客户端、新增客户端、客户端生命周期信息。

`analytics_clients` 只保存客户端生命周期和最近状态，不保存累计事件数。长期累计事件量统一从月度聚合表查询，避免回填与实时写入混合时客户端表计数难以纠错。

`reported_client_created_date` 来自客户端上报的 `client_created_at` 规范化结果，允许为空。`client_created_date` 是统计使用的有效创建日期，必须非空：如果客户端上报日期合法，则使用上报日期并将 `client_created_source` 设为 `reported`；如果缺失或非法，则使用首次上报的业务日期并将 `client_created_source` 设为 `first_seen`。新增客户端、排序和留存统计只使用 `client_created_date`，避免空字符串污染日期计算。

`first_seen_source`、`last_seen_source` 允许值为 `live` 或 `backfill`，用于回填纠错时判断某个客户端是否只来自回填数据。回填重跑只能删除 `last_seen_source = 'backfill'` 且最后活跃日期在回填覆盖范围内的客户端行；如果同一客户端后续有实时数据，`last_seen_source` 会变成 `live`，不得被回填清理删除。

### 长期唯一客户端计数表

月度事件数可以直接按月表求和，但“累计客户端数”不能把每月客户端数相加，否则同一个客户端跨月会重复。因此需要维度客户端去重表，并通过维度累计客户端表提供查询加速。

```sql
CREATE TABLE IF NOT EXISTS analytics_dimension_clients (
  project_name TEXT NOT NULL,
  dimension_type TEXT NOT NULL,
  dimension_key TEXT NOT NULL,
  client_id TEXT NOT NULL,
  first_seen_at TEXT NOT NULL,
  first_seen_date TEXT NOT NULL,
  first_seen_month TEXT NOT NULL,
  first_seen_source TEXT NOT NULL,
  last_seen_at TEXT NOT NULL,
  last_seen_date TEXT NOT NULL,
  last_seen_source TEXT NOT NULL,
  hit_count INTEGER NOT NULL DEFAULT 1,
  PRIMARY KEY (project_name, dimension_type, dimension_key, client_id)
);

CREATE INDEX IF NOT EXISTS idx_dimension_clients_lookup
ON analytics_dimension_clients (project_name, dimension_type, dimension_key);

CREATE INDEX IF NOT EXISTS idx_dimension_clients_backfill_cleanup
ON analytics_dimension_clients (project_name, last_seen_source, last_seen_date);
```

维度类型示例：

| dimension_type | dimension_key 示例 | 用途 |
| --- | --- | --- |
| event | `page_view` | 某类事件历史触达客户端数 |
| page | `technical-plan/content-edit` | 页面历史触达客户端数 |
| config | `minimumWords=1500` | 配置值历史触达客户端数 |
| model | `text|jinlong|jlaudeapi.com|gpt-4o` | 模型历史触达客户端数 |
| resource | `r_xxxxx` | 资源历史点击客户端数 |
| version | `0.1.0` | 版本历史触达客户端数 |

所有组合维度 key 必须生成稳定短 key，不只是在“可能较长”时才 hash。规则为：`dimension_key = <prefix>_<sha256(normalized-value).slice(0, 24)>`，原始规范化展示值写入 `analytics_dimension_values.label`。这样可以避免主键过长、索引膨胀和模型名、页面名变更导致的不可控 key 长度。

### 操作去重表

Queue 是至少一次投递。为避免 Consumer 重试导致月度计数重复，操作去重表是必选机制，不能作为可选优化。

```sql
CREATE TABLE IF NOT EXISTS analytics_processed_events (
  event_id TEXT PRIMARY KEY,
  received_date TEXT NOT NULL,
  status TEXT NOT NULL,
  locked_at TEXT NOT NULL,
  processed_at TEXT
);

CREATE INDEX IF NOT EXISTS idx_processed_events_received_date
ON analytics_processed_events (received_date);
```

该表只保存事件 ID、日期和处理状态，不保存事件明细。通过 Cron 定期删除 45 天前且 `status = 'done'` 的记录，使其保持有界。`status` 允许值为 `processing`、`done`。如果存在长时间停留在 `processing` 的记录，Consumer 可按 `locked_at` 超时规则重新接管。

## 写入链路

### `/track` Worker

1. 解析并规范化请求体。
2. 校验 `projectName`、`event`、`client_id`、`resource_key` 等字段。
3. 生成 `event_id`、`received_at`、`activity_date`、`month`、`shard`。
4. 先投递规范化事件到 `ANALYTICS_ROLLUP_QUEUE`。Queue 是长期统计的持久化入口，入队成功才表示长期统计链路已接收该事件。
5. Queue 入队成功后写入 Analytics Engine，保留现有近期查询和排查能力。
6. 返回 `202` 或其他 2xx 响应。
7. Queue 入队失败时，不写 Analytics Engine，返回 `503` 并输出 Worker 错误日志。客户端埋点调用仍然 fire-and-forget 并静默处理失败，不影响用户主流程。

禁止使用“先写 Analytics Engine，再投递 Queue”的顺序，否则会出现 Analytics Engine 已记录但 D1 长期统计未入队的半成功状态，后续无法可靠补偿。

Analytics Engine 写入失败只影响近期灵活分析和最近事件排查，不影响 D1 长期统计。只要 Queue 入队成功，`/track` 仍可返回 2xx，并记录 Analytics Engine 写入失败日志。

### Queue Consumer

Consumer 批量处理事件，单批建议 50 到 100 条。

Consumer 不允许按“每条事件、每个字段”直接写 D1。必须先在内存中完成批内聚合，再按聚合 key 分块 upsert，降低 D1 statement 数量和热点写入次数。

每条事件写入 D1 的逻辑：

| 步骤 | 表 | 操作 |
| --- | --- | --- |
| 1 | `analytics_processed_events` | 强制创建处理锁；`status = done` 时跳过，`processing` 超时后允许重新接管 |
| 2 | `analytics_clients` | upsert 客户端身份，更新首次/最近活跃、版本、平台和来源字段 |
| 3 | `analytics_daily_client_activity` | upsert 当日活跃行，累计当天事件数 |
| 4 | `analytics_monthly_event_stats` | 更新当月 `source = live` 事件桶，按 shard 累加 |
| 5 | 具体月度统计表 | 根据事件类型更新页面、版本、配置、模型、资源月度桶 |
| 6 | `analytics_dimension_clients` | 插入或更新维度客户端关系，用于历史累计客户端数 |
| 7 | `analytics_dimension_client_totals` | 仅当维度客户端关系首次插入成功时累计 `client_count` |
| 8 | `analytics_dimension_values` | 更新维度展示标签 |

Consumer 幂等规则：

1. `event_id` 必须由 Worker 在 `/track` 中生成并随 Queue 消息传递。
2. Consumer 先按 `event_id` 去重同一批消息，避免同一批内重复事件进入聚合。
3. 锁定阶段先尝试写入或接管 `analytics_processed_events`，只允许成功获得处理锁的事件进入聚合。
4. 已标记为 `done` 的 `event_id` 再次投递时必须直接跳过，不能再次累计。
5. 如果事件卡在 `processing` 且超过超时时间，新的 Consumer 可以重新接管；未超时的 `processing` 记录视为其他 Consumer 正在处理。
6. 聚合阶段把获得锁的事件按表和主键合并，例如同一个 `project + month + source + page + shard` 只生成一次月度页面 upsert。
7. 提交阶段按 statement 数量分块，每个块使用 D1 batch/事务提交；块内必须包含对应统计更新和 `status = done` 更新。
8. 如果任一块提交失败，Consumer 必须抛错让 Queue 重试，不得 ack 该批消息；失败事件保持 `processing`，后续通过超时接管重试。
9. 不允许绕过 `analytics_processed_events` 直接累计，也不允许先标记 `done` 再写统计。

批内聚合 key 示例：

| 聚合目标 | key |
| --- | --- |
| 月度事件 | `project_name + month + source + event + shard` |
| 月度页面 | `project_name + month + source + page + shard` |
| 月度版本 | `project_name + month + source + version + shard` |
| 月度配置 | `project_name + month + source + field_key + value + shard` |
| 月度模型 | `project_name + month + source + request_type + provider + endpoint_host + model + shard` |
| 月度资源 | `project_name + month + source + resource_key + shard` |
| 维度客户端 | `project_name + dimension_type + dimension_key + client_id` |

### 事件到月度表的映射

| event | 月度表 | 维度客户端 |
| --- | --- | --- |
| `app_open` | `analytics_monthly_event_stats`、`analytics_monthly_version_stats` | `event=app_open`、`version=<version>` |
| `page_view` | `analytics_monthly_event_stats`、`analytics_monthly_page_stats`、`analytics_monthly_version_stats` | `event=page_view`、`page=<page>`、`version=<version>` |
| `config_usage` | `analytics_monthly_event_stats`、`analytics_monthly_config_stats`、`analytics_monthly_version_stats` | `event=config_usage`、`config=<field=value>`、`version=<version>` |
| `ai_request` | `analytics_monthly_event_stats`、`analytics_monthly_model_stats`、`analytics_monthly_version_stats` | `event=ai_request`、`model=<request/provider/host/model>`、`version=<version>` |
| `resource_click` | `analytics_monthly_event_stats`、`analytics_monthly_resource_stats`、`analytics_monthly_version_stats` | `event=resource_click`、`resource=<resource_key>`、`version=<version>` |

## 性能策略

### 避免 D1 查询卡顿

1. 页面查询不扫原始事件，因为 D1 不存原始事件。
2. 历史总数只扫月度聚合表和 `analytics_dimension_client_totals`。
3. 今日、昨日、近 7/30 天活跃只查 `analytics_daily_client_activity` 的日期索引。
4. 长期总客户端只查 `analytics_clients` 的项目主键范围。
5. 排行查询必须带 `LIMIT`，默认最多返回 100 条。
6. 维度 key 保持低基数，不允许用户输入、文件名、本地路径、文档 ID、任务 ID 进入维度。
7. `analytics_dimension_clients` 只用于写入去重和离线排查，Dashboard 正常查询不得用它做实时 `COUNT(*)`。

### 避免 D1 写入热点

1. 月度计数表使用 `shard = hash(client_id) % 16`。
2. 统计查询时用 `SUM(count)` 聚合 16 个 shard。
3. Queue Consumer 批量写入，避免每个 `/track` 请求同步写多条 D1。
4. 对 `config_usage` 只写非空字段，避免一次配置上报产生过多无效维度。
5. 对模型维度只保存 `provider`、`endpoint_host`、`model`、`request_type`，不保存完整 URL。

### 数据量预估

| 表 | 增长方式 | 说明 |
| --- | --- | --- |
| `analytics_clients` | 每客户端 1 行 | 长期稳定，适合总客户端统计 |
| `analytics_daily_client_activity` | 每活跃客户端每天 1 行 | 例如 2000 日活约 73 万行/年，可通过日期索引查询 |
| 月度统计表 | 每月每维度每 shard 1 行 | 页面、模型、资源等维度数量有限 |
| `analytics_dimension_clients` | 每客户端每维度 1 行 | 用于历史客户端去重，不随事件次数线性增长 |
| `analytics_dimension_client_totals` | 每维度 1 行 | 页面查询累计客户端数时读取，避免扫描去重明细表 |
| `analytics_processed_events` | 每事件 1 行，但只保留 45 天 | 仅用于 Queue 重试去重，可定期清理 |

## 回填方案

新增脚本 `analytics/scripts/backfill-analytics-rollups.mjs`，用于从 Analytics Engine 尽量回填最近 90 天数据到 D1。

回填边界：

1. `rollup_started_at` 在 Queue 长期写入正式上线时固定写入 `analytics_meta`，后续不得自动更新。
2. 回填只覆盖 `rollup_started_at` 之前的业务日期，默认 `backfill_end_date = rollup_started_date - 1 day`，避免与实时 `source = live` 数据重叠。
3. 如果必须回填上线当天的部分数据，需要手工指定精确时间边界，并在脚本中按时间过滤，不能同时把同一事件计入 `backfill` 和 `live`。

回填范围：

| 数据 | 是否可回填 | 说明 |
| --- | --- | --- |
| 最近 90 天事件量 | 可以 | Analytics Engine 保留期内可查 |
| 最近 90 天页面排行 | 可以 | 按月聚合查询后写 D1 |
| 最近 90 天版本分布 | 可以 | 按月、版本聚合查询后写 D1 |
| 最近 90 天配置使用 | 可以 | 按月、字段和值聚合后写 D1 |
| 最近 90 天模型使用 | 可以 | 按月、模型组合聚合后写 D1 |
| 最近 90 天资源点击 | 可以 | 按月资源 key 聚合后写 D1 |
| 超过 90 天历史数据 | 不可恢复 | Analytics Engine 已过保留期的数据无法补回 |

回填幂等规则：

1. 月度聚合表使用 `source = backfill` 写入。
2. 每次重跑某个覆盖范围前，先删除该范围内所有 `source = backfill` 的月度聚合行，再按 Analytics Engine 查询结果插入绝对聚合值，不使用 `+` 累加。
3. 回填覆盖范围只允许在 `rollup_started_at` 之前，因此可以删除覆盖日期内的 `analytics_daily_client_activity` 后重建，避免按日计数残留。
4. `analytics_clients` 回填不累加事件计数，只修正生命周期字段；重跑前可删除 `last_seen_source = 'backfill'` 且 `last_seen_date` 落在覆盖范围内的 backfill-only 客户端行。
5. `analytics_dimension_clients` 回填使用主键去重，但重跑前必须删除 `last_seen_source = 'backfill'` 且 `last_seen_date` 落在覆盖范围内的 backfill-only 维度关系，避免旧回填结果残留。
6. 如果某个客户端或维度关系在回填后又被实时数据命中，`last_seen_source` 会变成 `live`，回填重跑不得删除该行。
7. 回填插入或更新维度关系后，必须从 `analytics_dimension_clients` 重算受影响 `project_name + dimension_type + dimension_key` 的 `analytics_dimension_client_totals`，不能用简单 `+1/-1` 维护。
8. 回填脚本必须可以重复执行，重复执行后的历史总数、累计客户端数、每日活跃不能发生翻倍，也不能保留已被纠正的旧维度关系。

回填重跑流程：

1. 计算 `coverage_start_date`、`coverage_end_date` 和覆盖月份列表。
2. 查询旧回填结果涉及的维度 key，记录为 `affected_dimensions`。
3. 删除覆盖月份内所有月度表的 `source = backfill` 行。
4. 删除覆盖日期内的 `analytics_daily_client_activity` 行。
5. 删除 `analytics_clients` 中 backfill-only 客户端行。
6. 删除 `analytics_dimension_clients` 中 backfill-only 维度关系，并把删除涉及的维度 key 加入 `affected_dimensions`。
7. 从 Analytics Engine 查询绝对聚合结果，写入 `source = backfill` 的月度表。
8. 从 Analytics Engine 查询客户端日活、客户端生命周期和维度客户端关系，重建每日活跃、客户端生命周期和维度关系。
9. 把新写入涉及的维度 key 加入 `affected_dimensions`。
10. 对 `affected_dimensions` 逐个从 `analytics_dimension_clients` 重新计算 `client_count`、`first_seen_at`、`last_seen_at`，并覆盖写入 `analytics_dimension_client_totals`；如果某个维度 key 已没有任何客户端关系，则删除对应 totals 行。
11. 写入 `analytics_meta` 中的回填覆盖范围和完成时间。

回填完成后写入 `analytics_meta`：

| key | value |
| --- | --- |
| `rollup_started_at` | 长期统计正式开始时间 |
| `backfill_coverage_start_date` | 回填覆盖起始日期 |
| `backfill_coverage_end_date` | 回填覆盖结束日期 |
| `backfill_completed_at` | 最近一次回填完成时间 |

Dashboard 长期累计区域需要显示类似说明：

```text
长期累计自 2026-xx-xx 起持续记录，已回填 2026-xx-xx 之后的 Analytics Engine 可查询数据。
```

## 安全与隐私

1. 不新增任何用户个人身份信息采集。
2. D1 只保存已有埋点里的匿名 `client_id`。
3. 不保存 `api_key`、完整 `base_url`、`mineru_token`、Prompt、AI 响应、错误详情、文件名、本地路径、招标文件内容、正文草稿。
4. 模型接口只保存 `endpoint_host`，不保存完整 URL。
5. 页面、资源、配置、模型维度必须是稳定低基数字段，不允许用户输入或随机 ID 进入维度。
6. Dashboard 的 ADMIN_TOKEN 默认只放 sessionStorage，关闭浏览器后失效；用户显式选择“记住 Token”才写 localStorage。
7. Dashboard 增加明确提示：记住 Token 会提高本机浏览器持久化风险，只适合受信任设备。
8. Dashboard 页面增加 CSP，默认禁止第三方脚本，`connect-src` 只允许 `self` 和 `https://analytics.agnet.top`；开发环境如需连接本地 Worker，通过开发专用配置追加允许域名。

## 风险与处理

| 风险 | 处理 |
| --- | --- |
| Queue 至少一次投递导致重复计数 | 强制使用 `analytics_processed_events` 做事件 ID 去重和处理锁 |
| Consumer 单批写入语句过多 | 先按统计 key 做批内聚合，再按 statement 数量分块提交 D1 batch |
| D1 月度计数热点行 | 月度表使用 16 shard，查询时汇总 |
| 历史唯一客户端数跨月重复 | 使用 `analytics_dimension_clients` 做长期去重，不累加每月客户端数 |
| 历史累计客户端数查询慢 | 使用 `analytics_dimension_client_totals` 作为查询入口，不实时扫描维度客户端明细表 |
| Analytics Engine 回填不完整 | Dashboard 显示回填覆盖起点，超过保留期的数据明确不可恢复 |
| 回填脚本重复执行导致翻倍 | 月度表按 `source = backfill` 删除后重建，其他表使用绝对值 upsert 或主键去重 |
| 配置或模型维度过多 | 只写稳定低基数字段，字段值长度限制和规范化 |
| D1/Queue 未配置导致长期数据丢失 | Worker 部署前自动检查并配置，生产环境不做静默降级 |
| ADMIN_TOKEN 持久化带来 XSS 风险 | 默认 sessionStorage，localStorage 需要用户显式选择，并通过 CSP 和 HTML 转义降低风险 |

## 非目标

1. 不把所有原始事件长期保存到 D1。
2. 不上传或保存任何敏感业务内容。
3. 不把 Analytics Engine 完全移除，它仍用于近期事件和灵活分析。
4. 不尝试恢复 Analytics Engine 保留期之外的历史数据。
